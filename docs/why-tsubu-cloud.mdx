# Why tsubu-cloud

Cloud Run、ECS/Fargate、Heroku、Railway といった CaaS の上で普通に Web サービスを
動かすとき、機能とは関係のないところで確実に効いてくるコストがあります。ここでは
その代表的な2つを、CaaS の現実と tsubu-cloud での解消の対比で見ていきます。

## シナリオ1: HTTP を受ける層のバージョン管理が、ずっと自分の責任になる

### CaaS の現実

CaaS が動かすのはあなたのコンテナ(あるいはリクエストハンドラ)です。HTTP
リクエストを受けてパースし、ルーティングしてハンドラに渡す層は、アプリ側で
自分で用意しなければなりません。実際には Hono・Express・Next.js・axum といった
Web フレームワークを入れることになります。

問題は、それが一度入れて終わりではない点です。

- HTTP を喋るための足回りが、アプリ本来のロジックとは無関係に依存ツリーへ常駐します
- CVE が出れば追随し、メジャーバージョンの破壊的変更も自分で吸収し続ける必要があります
- この保守責任は、サービスを運用している限り消えません

アプリがやりたいのは「リクエストを受けてレスポンスを返す」ことなのに、その
「受けて返す」ための土台の面倒を、ずっと見続けることになります。

### tsubu-cloud での解消

tsubu-cloud のゲストが書くのは、次の一つの関数の実装だけです。

```wit
export handler: func(request: request) -> response;
```

```moonbit
pub fn handler(_request : @types.Request) -> @types.Response {
  @types.Response::{ status: 200, headers: [], body: "Hello World" }
}
```

HTTP を受け付けてパースし、`request` に変換してハンドラを呼ぶ層は、ホスト
(tsubu-cloud-zig)が持っています。アプリは Web フレームワークをバンドルしません。

- HTTP を受ける層の依存とバージョン管理が、アプリのコードベースから消えます
- その土台はホスト側で更新され、すべてのゲストに反映されます
- アプリに残るのは、`request` を受けて `response` を返す純粋なロジックだけです

:::note
アプリのビジネスロジックが持つ依存が消えるわけではありません。消えるのは
「HTTP を喋るための足回り」の保守責任です。
:::

## シナリオ2: outbound 通信をドメイン単位で止められない

### CaaS の現実

数百の依存パッケージの上に成り立つ普通の Web サービスでは、その中の1つが侵害
される、あるいはただバグっているだけでも、コード内のどこからでも外部にデータを
送信できてしまいます。サプライチェーン攻撃、SSRF、意図しない秘密情報の漏洩です。

これを防ぐ標準的な手段は「outbound 通信の制限」ですが、CaaS では実際には
まともに機能しません。

- 制限自体は VPC の egress firewall や NAT/プロキシで可能ですが、これは IP
  アドレスベースのフィルタリングです。正規の通信先が外部 SaaS や CDN 配下の
  API など動的 IP の場合、宛先を特定できずレンジを広く開けざるを得ません。
  結果として制限が形骸化するか、運用が面倒でそもそもやらないことになります
- Cloudflare Workers のように、outbound 通信を制限する仕組みがそもそも
  ないプラットフォームもあります
- Spin など一部の WASM ランタイムはホスト名ベースの allowlist を持ちますが、
  コード側は汎用の `fetch` を持ったまま実行時にホスト名をチェックするだけです。
  設定ファイルとコードが二重管理になり、更新漏れ・許可の緩め過ぎが典型的な
  事故要因になります

共通するのは、制限が実行時フィルタとしてコードの外側に後付けされている点です。
フィルタである以上、設定を書き忘れれば通ってしまい(fail-open)、コードは
制限されていない通信先を呼び出す能力自体を持ち続けます。

### tsubu-cloud での解消

tsubu-cloud では、[Core Concepts](/core-concepts) の通り、`tsubu.json` に宣言した
通信先だけが WASM Component の import として存在します。

```json
{
  "$schema": "https://tsubu-cloud.net/tsubu.schema.json",
  "fetch": [
    { "ALIAS": "fetch", "TARGET": "https://example.com" }
  ],
  "postgres": [],
  "variables": []
}
```

宣言していない通信は、実行時フィルタで弾かれるのではなく、コンポーネントの
構造としてそもそも存在しません。

- **単位が IP ではなくエイリアス**: 正規の通信先が動的 IP でも、宛先を広く
  開ける必要がありません
- **インフラ側の別設定が不要**: VPC・NAT・Firewall を整備しなくても、ホスト側の
  linker 設定だけで完結します
- **fail-closed**: import されていない capability は呼び出す関数自体が生成
  されないので、「制限を忘れる」ことができません。設定(`tsubu.json`)と
  コード(`wit/guest.wit` から生成されたバインディング)が同じ WIT 定義から
  導出されるため、ズレが構造的に起き得ません
- **監査可能性**: 何が呼べるかは `tsubu.json` と `wit/guest.wit` を見るだけで
  完全にわかります。ランタイムの挙動を追ったり、依存パッケージの中身まで
  読む必要がありません

## どんな人向けか

特別なセキュリティ要件を持つ層に限りません。普通に Web サービスを書いていて、
依存パッケージの数と信頼できなさに漠然とした不安がある開発者全般が対象です。

- 「うちのアプリ、依存の中身まで全部信用して外部通信を許可してるんだっけ?」
  という不安から解放されます
- デプロイ設定ファイル一つが、そのままそのアプリの通信先の完全な一覧になります
- コードレビューや監査の手間も減ります
