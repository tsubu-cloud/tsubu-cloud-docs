# Design Rationale

## 課題

普通の Web サービス開発は、数百の依存パッケージの上に成り立っています。その中の
1つが侵害される、あるいはただバグっているだけでも、コード内のどこからでも
外部にデータを送信できてしまいます — サプライチェーン攻撃、SSRF、意図しない
秘密情報の漏洩です。

これを防ぐ標準的な手段は「outbound 通信の制限」として確立していますが、実際には
まともに機能していません。

- **Cloudflare Workers 等**: outbound 通信を制限する仕組みがそもそもありません
- **一般的な CaaS(Cloud Run, ECS/Fargate, Heroku, Railway など)**: 制限自体は
  VPC の egress firewall や NAT/プロキシで可能ですが、これは IP アドレスベースの
  フィルタリングです。正規の通信先が外部 SaaS や CDN 配下の API など動的 IP の
  場合、宛先を特定できずレンジを広く開けざるを得ません。結果として制限が
  形骸化するか、そもそも運用が面倒でやらないことになります
- **Spin 等の一部 WASM ランタイム**: ホスト名ベースの allowlist の仕組みは
  ありますが、コード側は汎用の `fetch` import を持ったままで、実行時にホスト名を
  チェックするだけです。設定ファイルとコードが二重管理になり、更新漏れ・許可の
  緩め過ぎが典型的な事故要因になります

共通しているのは、「制限」が実行時のフィルタとしてコードの外側に後付けされて
いる点です。フィルタである以上、設定を書き忘れれば通ってしまいますし
(fail-open)、コードは制限されていない通信先を呼び出す能力自体は持ち続けます。

## 解決

tsubu-cloud では、[Core Concepts](/core-concepts) で説明した通り、`tsubu.json`
に宣言した通信先だけが WASM Component の import として存在します。宣言して
いない通信は実行時フィルタで弾かれるのではなく、コンポーネントの構造として
そもそも存在しません。

これは他のアプローチと比べて次の点が異なります。

- **単位が IP ではなくエイリアス**: 正規の通信先が外部 SaaS や CDN 配下の
  動的 IP でも、宛先を広く開ける必要がありません
- **インフラ側の別設定が不要**: VPC・NAT・Firewall の整備をしなくても、
  ホスト側の linker 設定だけで完結します
- **実行時フィルタへの信頼が不要**: 設定(`tsubu.json`)とコード
  (`wit/guest.wit` から生成されたバインディング)が同じ WIT 定義から
  導出されるため、ズレが構造的に起き得ません。フィルタの設定漏れという
  典型的な事故パターンそのものが存在しません
- **fail-closed**: import されていない capability は呼び出す関数自体が
  生成されないので、「制限を忘れる」ことができません
- **監査可能性**: 何が呼べるかは `tsubu.json` と `wit/guest.wit` を見るだけで
  完全にわかります。ランタイムの挙動を追ったり、依存パッケージの中身まで
  読む必要がありません

## 効き方

「うちのアプリ、依存の中身まで全部信用して外部通信を許可してるんだっけ?」と
いう漠然とした不安から解放されます。

デプロイ設定ファイル一つが、そのままそのアプリの通信先の完全な一覧になります。
コードレビューや監査の手間も減ります。

## ターゲット

特別な要件を持つ層に限りません。普通に Web サービスを書いていて、依存
パッケージの数と信頼できなさに漠然とした不安がある開発者全般が対象です。
